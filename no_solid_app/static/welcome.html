No SOLID Bank API

A banking application built with FastAPI deliberately ignoring SOLID principles

About This Application

This banking API was developed as part of a research project to evaluate the impact of design patterns without SOLID principles. The application implements core banking functionalities including:

  • User authentication and management
  • Account creation and management
  • Financial transactions (deposits, withdrawals, transfers)
  • Transaction history

This version deliberately ignores SOLID principles while still using design patterns, to demonstrate the differences in code quality, maintainability, and extensibility.

Design Patterns Used

Singleton Pattern

Violates SRP Violates OCP Violates DIP

Used for user management and authentication, forcing all user-related functionality into a single class.

This creates a god object with multiple responsibilities, making changes difficult and increasing the risk of side effects.

Facade Pattern

Violates SRP Violates OCP Violates DIP

Handles all transaction types through a single facade that coordinates multiple operations.

While this simplifies the client interface, it creates a monolithic component that's difficult to extend without modifying existing code.

Abstract Factory Pattern

Violates ISP Violates DIP

Creates different types of accounts but forces concrete factories to implement interfaces they might not need.

This violates the Interface Segregation Principle by making classes depend on methods they don't use.

SOLID Principles Violations

  • Single Responsibility Principle: Classes have multiple responsibilities. For example, the Singleton auth_manager handles both authentication and session management.
  • Open/Closed Principle: The application requires modification of existing code to add new features, like adding a new transaction type to the Facade.
  • Liskov Substitution Principle: Subtypes might not maintain the behavior of their parent types, potentially causing unexpected issues.
  • Interface Segregation Principle: Classes are forced to implement interfaces they don't use, especially with the Abstract Factory pattern.
  • Dependency Inversion Principle: High-level modules depend directly on low-level modules, creating tight coupling and making the system less flexible.

Research Implications

This application deliberately demonstrates how design patterns alone, without adherence to SOLID principles, can lead to:

  • Increased complexity and code size
  • Reduced maintainability
  • Higher coupling between components
  • Greater difficulty in extending functionality
  • More challenging testing scenarios

The comparison with the SOLID-compliant version shows the value of combining design patterns with sound architectural principles.
