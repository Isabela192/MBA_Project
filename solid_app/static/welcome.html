SOLID Bank API

A banking application built with FastAPI following SOLID principles and design patterns

About This Application

This banking API was built as part of a research project to evaluate the impact of applying SOLID principles in conjunction with design patterns. The application implements core banking functionalities including:

  • User management (clients and managers)
  • Account creation and management
  • Financial transactions (deposits, withdrawals, transfers)
  • Transaction history

All of these features are implemented using SOLID principles to ensure clean, maintainable, and extensible code.

Design Patterns Used

Factory Method

S - Single Responsibility O - Open/Closed L - Liskov Substitution D - Dependency Inversion

Used for creating different types of users (clients and managers) through specialized factories that handle the creation logic.

Each factory is responsible only for creating a specific type of user, making the system more modular and easier to extend.

Command Pattern

S - Single Responsibility O - Open/Closed I - Interface Segregation D - Dependency Inversion

Each financial operation (deposit, withdraw, transfer) is encapsulated as a separate command object.

This makes it easy to add new transaction types without modifying existing code, and each command has a single responsibility.

Proxy Pattern

S - Single Responsibility O - Open/Closed

The AccountProxy controls access to account operations, adding a layer of validation and security.

This separation allows modifications to access control logic without changing the core account functionality.

SOLID Principles Applied

  • Single Responsibility Principle: Each class has one job. For example, the Command pattern ensures that each transaction type has its own class with a single responsibility.
  • Open/Closed Principle: The application is open for extension but closed for modification. New transaction types can be added without modifying existing code.
  • Liskov Substitution Principle: Subtypes (like different user factories) can be substituted for their base types without affecting behavior.
  • Interface Segregation Principle: Commands implement only the methods they need, avoiding unnecessary dependencies.
  • Dependency Inversion Principle: High-level modules depend on abstractions, not concrete implementations, making the system more flexible.
